## 1. 产品定义

这个项目不应该被定义成一个普通的低代码页面编辑器，而应该被定义成：

**一个面向非技术用户的“前端体验编排器 + AI 交付桥接层”。**

目标不是让用户自己把整套系统从前到后都做完，而是让用户先把：

- 页面结构
- 模块布局
- 交互意图
- 数据占位
- 组件层级
- 响应式关系

以可视化方式快速搭出来，然后把这份结构化产物稳定交给 AI Coding，让 AI 去完成：

- 前端真实代码落地
- 后端接口对接
- 数据查询与提交
- 权限与鉴权
- 部署与联调

换句话说，这个项目的核心价值不是“帮用户画页面”，而是：

**把模糊需求变成 AI 不容易误解的结构化前端协议。**

## 2. 核心用户与场景

### 核心用户

- 懂业务但不懂前端工程的产品/运营/老板
- 会描述页面，但很难准确表达给开发或 AI 的业务用户
- 需要快速做 dashboard、门户、活动页、表单页、后台页面的团队

### 典型场景

1. 用户要做一个官网或活动页
2. 用户要做一个一屏 dashboard 或业务工作台
3. 用户先拖拽出页面和模块
4. 用户设置每个模块的标题、文案、尺寸、排列、交互动作、数据来源意图
5. 系统输出结构化页面协议
6. AI Coding 根据协议一次性补齐真实前端代码和后端接口
7. 最终端到端打通

## 3. 产品边界

### 这个项目应该做的

- 可视化编排页面结构
- 支持组件拖拽、嵌套、布局和尺寸调整
- 支持属性编辑、动作绑定、数据绑定
- 支持草稿、版本、模板和复用
- 支持导出机器可读的页面协议
- 支持把协议交给 AI 生成代码与后端接入方案

### 这个项目暂时不应该做的

- 一开始就做完整 CMS
- 一开始就做超复杂权限系统
- 一开始就做全量站点托管平台
- 一开始就支持太多异构后端连接器

## 4. 系统分层

建议按 5 层搭建，而不是直接堆页面编辑功能。

### A. Builder 编辑层

面向用户的可视化编辑器：

- 左侧组件面板
- 中央画布
- 右侧属性面板
- 页面/路由管理
- 响应式断点管理
- 模板库

### B. Schema 协议层

这是最关键的一层，决定 AI handoff 是否稳定。

必须有严格定义的结构化协议：

- `ProjectSchema`
- `PageSchema`
- `NodeSchema`
- `LayoutSchema`
- `PropSchema`
- `BindingSchema`
- `ActionSchema`
- `DataSourceSchema`

协议层职责：

- 统一保存格式
- 统一导出格式
- 校验用户编辑结果是否合法
- 给 AI 提供稳定、低歧义输入

### C. Runtime 渲染层

同一份 schema 必须支持三种模式：

- `edit`：编辑态
- `preview`：预览态
- `runtime`：真实运行态

也就是说，编辑器不能只是一堆假的占位块，而要能逐步过渡到真实渲染。

### D. AI Bridge 层

这层专门负责把 schema 转成 AI 容易执行的任务描述：

- 生成组件树摘要
- 生成接口需求清单
- 生成页面行为说明
- 生成数据绑定说明
- 生成后端 API 合同草案
- 生成前端待实现清单

最终目标不是输出“自然语言描述”，而是输出：

- 结构化 JSON
- 代码生成上下文
- 任务拆解清单
- 可校验的接口契约

### E. Integration 集成层

用于连接真实服务：

- REST API
- GraphQL
- 静态 JSON
- Mock 数据
- 鉴权信息
- 环境变量

这层一开始只做最小集合：

- REST 请求
- GET/POST
- Header/Token
- Mock Response

## 5. 数据模型建议

建议尽快把当前状态管理收敛为规范数据模型。

### Workspace / Account

这层要尽早存在，即便第一阶段先用本地模式模拟：

- 一个用户账户可以拥有多个工作区
- 一个工作区可以保存多个前端项目
- 当前活跃工作区和当前活跃项目必须分离保存
- 项目列表默认按工作区隔离，而不是所有项目混在一起

建议产品分两步落地：

1. **Local Workspace 模式**
   - 先在本地 IndexedDB 中实现工作区隔离
   - 支持切换工作区、每个工作区独立保存项目
   - 这一步解决“一个用户管理多个前端方案”的问题
2. **Cloud Account 模式**
   - 后续再接真实登录、云端存储、团队协作
   - 届时把本地工作区映射成账户下的 workspace / tenant
   - 这样不会推翻当前底座

工程上建议同步保留一层 **Session / Workspace Gateway**：

- 上层 Store / UI 只依赖统一网关接口
- 当前先接 `localWorkspaceGateway`
- 后续可替换成 `cloudWorkspaceGateway`
- 这样认证、活跃工作区、项目索引、草稿保存都能无缝切换

### Project

- 项目元信息
- 页面列表
- 组件注册表版本
- 数据源列表
- 环境配置

### Page

- 页面 ID
- 页面名称
- 路由
- 布局树根节点
- SEO / 页面设置

### Node

- 稳定 ID
- 组件类型
- 属性
- 布局信息
- 子节点
- 数据绑定
- 动作绑定
- 可见性规则

### Binding

- 数据来源
- 字段映射
- 默认值
- 空态配置
- 错误态配置

### Action

- 点击行为
- 跳转
- 打开弹窗
- 提交请求
- 刷新数据
- 调用脚本

## 6. MVP 应该怎么切

### MVP-1：前端体验编排

先完成：

- 页面画布
- 组件拖拽
- 嵌套容器
- 属性编辑
- 本地草稿
- 模板复用

### MVP-2：结构化导出

补上：

- 页面 schema 导出
- schema 校验
- 版本化存储
- 导入/导出

### MVP-3：数据与交互

补上：

- Mock 数据源
- 组件字段绑定
- 基础动作系统
- 预览态

### MVP-4：AI Handoff

补上：

- 导出给 AI 的任务包
- 组件树说明
- API contract 草案
- 前后端联调说明

### MVP-5：真实后端打通

补上：

- REST Connector
- 认证配置
- 运行态调用
- 页面结果验证

### MVP-6：账户与工作区隔离

补上：

- 用户/工作区/项目三级关系
- 工作区切换与项目隔离
- 当前工作区活跃项目记忆
- 为未来云端账号体系预留可替换接口

## 7. 当前仓库现状

当前仓库已经不只是“编辑器雏形”，而是已经具备一条可继续收敛的主链路：

- Builder 壳层、Page Board、页面 / Overlay 拓扑与多面板工作台
- `ProjectDocument` / `ProjectBundle` / schema 校验 / 导入导出 / 版本切换
- 本地工作区、多项目隔离、草稿与快照存储
- `local / cloud / supabase` 可替换工作区网关骨架
- 运行态页面渲染骨架、数据源协议、基础动作 / 绑定结构
- 主题 preset、reference profile、导入编译与项目主题作用域
- AI handoff 包、导出就绪校验与前端交付包生成
- `Mock Cloud` 服务与页面拓扑校验脚本

因此当前缺口已经从“有没有这层”切换成“边界是否稳定、工程是否可持续”。

当前主要问题集中在以下几类：

1. **工程拆分不足**
   - `src/pages/BuilderPage.tsx`
   - `src/store/builderStore.ts`
   - `src/components/PageBoard.tsx`
   - `src/components/ProtocolPanels.tsx`
   - `src/components/BuilderShellPanels.tsx`
   这些文件已经承载过多职责，后续修复如果继续直接堆加逻辑，维护成本会迅速上升。

2. **拓扑与 Overlay 边界还需要回归**
   - 页面 / Overlay 的尺寸约束、宿主页同步、轨道排布已经有实现
   - 但当前 `scripts/verify-page-topology.ts` 仍能打出断言偏差，说明这部分需要优先进入下一轮修复

3. **云端链路仍以骨架为主**
   - 本地模式已经能完成主要开发
   - Cloud / Supabase 网关已预留替换接口
   - 但真实鉴权、稳定同步、异常恢复还没有完全走通

4. **自动化回归不足**
   - 当前已有类型检查与少量脚本化校验
   - 但围绕拓扑、导出、运行态、版本切换的自动化验证仍不够密

5. **编辑态 / 预览态 / 运行态仍需继续解耦**
   - 当前已经共享协议与部分运行层
   - 但壳层逻辑、编辑逻辑、运行逻辑之间还有继续下沉和隔离的空间

## 8. 工具选择原则

本项目更适合“轻量、可控、结构化”的工具，而不是大而全平台。

已安装的基础工具：

- `zod`：定义和校验 schema
- `dexie`：浏览器本地版本化存储
- `nanoid`：稳定节点 ID
- `zundo`：撤销 / 重做
- `vite-plugin-checker`：开发期诊断

## 9. 推荐实施顺序

### Phase 1：先修边界回归

- 修复 `Page Board` / `Overlay` 尺寸同步与拓扑校验偏差
- 对齐 `WebStageFrame`、`responsive`、`pageTopology` 的单一约束源
- 把当前失败的校验脚本变成稳定回归入口

### Phase 2：再拆工程职责

- 拆分 `BuilderPage` 的页面壳编排职责
- 拆分 `builderStore` 的协议转换、持久化编排与页面编辑逻辑
- 拆分 `PageBoard`、`ProtocolPanels`、`BuilderShellPanels` 的复合 UI 职责

### Phase 3：补运行态与导出闭环

- 继续收敛 edit / preview / runtime 的共享协议边界
- 补强导出就绪校验、前端交付包与 AI handoff 的一致性验证
- 把版本切换、导入覆盖、发布版本作为固定回归场景

### Phase 4：稳定数据与云端网关

- 继续补齐 REST / Mock 数据请求的运行态验证
- 收敛 Cloud / Supabase 网关的鉴权、同步和异常恢复路径
- 让本地模式与云端模式维持同一套上层接口

### Phase 5：增加脚本化回归密度

- 为页面拓扑、导出、运行态、版本切换补最小自动化脚本
- 避免后续修一个点、退两个点
- 让每轮修复都有明确的完成判据
- 暂缓处理的 debug 项统一登记在 `docs/debug-record.md`，待正式开修时再补维修方案
- 发现问题、开始修复、修复完成时，均要回写 `docs/debug-record.md`，形成持续更新的维修记录

## 10. 判断项目是否成功的标准

不是“页面能拖出来”，而是下面这件事是否成立：

> 一个非技术用户可以在 10~20 分钟内搭出页面草图；
> AI 能基于导出的结构化协议，稳定完成真实代码和后端打通；
> 人只需要做很少修正。

如果能做到这一点，这个项目就不是普通拖拽编辑器，而是：

**AI 时代的前端需求编译器。**

## 11. 账户隔离与多项目策略

从用户视角，这个产品必须支持：

- 一个用户同时保存多个前端项目
- 在多个项目之间切换，而不是反复覆盖同一份草稿
- 后续把不同项目交给不同 AI 任务流或后端服务去接入

因此骨架期就要先把“单项目草稿”升级成：

- `Workspace -> Projects -> Pages`

但骨架期不必一次性做完整登录系统。更合理的做法是：

1. **现在就完成本地多项目模型**
   - 本地工作区
   - 项目索引
   - 当前激活项目
   - 每个项目独立草稿 / 快照
2. **后续再接真实账户体系**
   - 登录 / 注册
   - 云端同步
   - 成员协作
   - 权限控制

也就是说：

- **多项目隔离：骨架期必须做**
- **完整账户认证：骨架期可以预留接口，不必一次做完**
