# 组件进货与母版接入规则 v1

## 1. 目的

从当前阶段开始，平台不再优先自研“看起来像组件”的资产，而是优先引入：

**社区里已经验证过的、适合纳入标准导出栈的优秀 React 组件。**

但这里的“引入”不是直接堆库，而是：

**把优秀组件吸收为母版能力，再编译进用户导出系统。**

这份文档规定：

- 什么组件可以进货
- 什么组件不该进货
- 进货后如何接入平台语法
- v1 第一批应该先引入哪些类型

准入后的正式验收标准，统一以 `docs/kit-contract-v1.md` 为准。

---

## 2. 核心原则

### 原则一：优先源码型，不要黑盒型

优先选择能直接以源码方式加入项目的组件体系，而不是重依赖、黑盒封装、难以控制导出结构的 UI 库。

原因：

- 我们最终要导出用户项目代码
- 用户导出物不能强依赖不透明组件黑盒
- AI Coding 必须能读懂、修改、延展这些组件

### 原则二：优先服务导出，而不是服务编辑器炫技

组件进货不是为了让左栏看起来很丰富，而是为了让用户导出结果更稳定、更标准、更容易接后端。

### 原则三：优先 React + TypeScript 主流生态

所有母版组件都必须兼容：

- React
- TypeScript
- Vite
- Tailwind CSS
- token 化主题

### 原则四：优先可被“卡片语法”吸收

外部组件进入平台后，不能保留自己的混乱表达方式，必须转换进我们的统一语法：

- `card shell`
- `control slots`
- `theme tokens`
- `bindings`
- `actions`
- `ai handoff`

### 原则五：少量、多轮、验证后再扩货

不追求一口气进几十个组件。  
先少量进货，验证：

- 视觉效果
- 可组合性
- 导出一致性
- AI handoff 是否清晰

通过后再扩展。

---

## 3. 组件源优先级

### A 级：优先采用

#### `shadcn/ui`

原因：

- 源码型组件体系
- 与 React + Tailwind + TypeScript 高度一致
- 社区成熟
- 非常适合作为导出代码的基础母体

适合承载：

- Button
- Card
- Input
- Dialog
- Tabs
- Table
- Select
- Calendar
- Chart 包装层

官方文档：

- `https://ui.shadcn.com/docs/installation/vite`
- `https://ui.shadcn.com/docs/installation`

#### `React DayPicker`

原因：

- 日期与日历能力成熟
- 已被 `shadcn Calendar` 采用
- 适合做日历卡片、日期选择卡片、内容归档日历

官方文档：

- `https://daypicker.dev/start`

#### `Tremor`

原因：

- 非常适合 dashboard / 数据展示类卡片
- 指标、趋势、图表、列表类表现成熟
- 很适合作为 card 模板来源

适合承载：

- KPI 卡片
- Trend 卡片
- Bar List
- Area Chart
- Spark Chart

官方文档：

- `https://www.tremor.so/`
- `https://www.tremor.so/docs/getting-started/installation`

#### `TanStack Table`

原因：

- 表格能力成熟
- 适合导出后继续接后端
- 排序、筛选、分页都标准化

官方文档：

- `https://tanstack.com/table/docs`

### B 级：可选引入

- `Recharts`
- `Embla Carousel`
- `cmdk`

这些可以作为局部能力来源，但不是 v1 第一优先级。

### C 级：暂缓

- 富文本编辑器
- 地图
- 全功能排程
- 文件上传管理器
- 重型低代码表单生成器
- 看板/甘特/流程引擎

这些复杂件很容易破坏 v1 的语法边界。

## 4. 平台内部统一语法

任何外来组件进入平台后，都不能直接裸露给用户。

它必须落入以下统一层级：

### `Page Shell`

路由级边界，不承载外来组件细节。

### `Card Shell`

所有外来可复用 UI 模板，最终都必须归入 card shell。

### `Controls`

卡片内部元素必须能拆解成 control 级内容单元，或至少暴露明确的 slot。

### `Bindings`

所有有数据来源的组件，必须暴露绑定入口，而不是把 mock 数据写死。

### `Actions`

所有有交互行为的组件，必须暴露动作入口。

### `AI Handoff`

每个可导出的卡片模板，都必须能附带一段实现意图说明，方便 AI Coding 接手。

---

## 5. 外来组件接入的五步法

### 第一步：选型

确认组件来源是否满足：

- React + TS
- 源码可接入
- 主题可 token 化
- 非黑盒强依赖

### 第二步：拆壳

把组件还原成：

- 壳体结构
- 内容区
- 可替换 slot
- 数据字段
- 动作入口

### 第三步：语法映射

映射进我们的平台语法：

- card shell
- controls
- bindings
- actions
- handoff

### 第四步：母版化

作为可拖入的标准资产出现在左侧资产区。

### 第五步：导出验证

检查它导出后是否仍然：

- 结构清晰
- 类型清晰
- 主题清晰
- 数据入口清晰
- AI 可接手

只要某个组件在这一步失败，就不能正式进入资产库。

## 6. v1 第一批进货范围

v1 只允许先引入以下少量高频资产：

### 1. `Card Shell Base`

作用：

- 所有卡片的基础壳
- 不承载业务预设
- 是用户自己组装 kit 的起点

### 2. `Calendar Card`

来源建议：

- `shadcn Calendar`
- `React DayPicker`

用途：

- 内容归档
- 日期导航
- 日期选择
- dashboard 时间视图

### 3. `KPI Stat Card`

来源建议：

- `Tremor`
- 或等价成熟 metric card 设计

用途：

- 指标数字
- 同比/环比
- 简短趋势说明

### 4. `Trend Chart Card`

来源建议：

- `Tremor`
- `shadcn Chart`

用途：

- 折线
- 面积图
- 小型趋势图

### 5. `Data Table Card`

来源建议：

- `shadcn Data Table`
- `TanStack Table`

用途：

- 列表
- 数据查询
- 表格展示

---

## 7. v1 不应立即进货的内容

以下组件虽然“看起来很强”，但现在不应优先引入：

- 富文本编辑器
- Markdown 编辑器
- 文件管理器
- 地图定位组件
- 拖拽看板
- 甘特图
- 多维度数据透视表
- 复杂图形白板

这些东西一旦提前引入，会让平台语法先失控。

---

## 8. 对左侧资产区的直接约束

在正式接入新组件前，左侧资产区必须继续保持克制。

原则：

- 不因为“有组件”就立即上架
- 必须先过语法映射
- 必须先通过导出验证
- 必须能解释清楚它在用户导出工程中的落点

如果说不清：

- 它属于 page
- 属于 card
- 属于 control
- 属于 kit

那它就不能进资产区。

---

## 9. v1 当前执行策略

接下来组件进货按这个顺序执行：

1. 先确认源库与准入条件
2. 先做一个接入样板
3. 放入平台内部验证
4. 验证通过后，再成为正式母版资产
5. 再用这些资产去做第一套 blog blueprint

---

## 10. 一句话总结

从现在开始，平台资产建设遵循的不是“想到什么做什么”，而是：

**优先吸收成熟 React 社区组件，经过统一语法改造后，变成可导出、可接后端、可交给 AI Coding 的标准母版资产。**
