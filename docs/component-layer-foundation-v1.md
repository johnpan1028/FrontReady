# 组件分层底层规范 v1

## 0. 目的

这份文档用于固化 `Kit Studio` 后续开发时，关于 **组件应该拆到哪一层、什么该暴露给用户、什么只做系统内置原语** 的统一规则。

它解决三个核心问题：

1. 开发时如何稳定搭建组件体系，不把职责做乱
2. 用户实际组装时，什么粒度最顺手、效率最高
3. 后续从 `shadcn`、`Tremor`、`TanStack Table` 等外部来源导入成品卡时，应该怎样拆壳、拆层、拆到位

这份文档是：

- `Kit Studio` 的组件分层基线
- 右侧属性栏分组的前置依据
- 外来组件适配模块的拆解标准
- 后续 AI / Agent 批量导入组件时的判断规则

## 2. 总原则

组件分层不按“代码里有几个 DOM 节点”来定，而按以下三条来定：

### 2.1 业务语义优先

如果一个东西离开宿主后，不能独立表达用户想组装的业务意义，它就不应该成为一个画布控件。

例如：

- 关闭按钮
- 拖动柄
- 展开箭头
- 勾选框里的小对勾

这些都有 UI 作用，但没有独立业务语义，因此不应进入左侧栏。

### 2.2 用户编辑效率优先

如果一个结构拆得太细，会让用户：

- 选中次数变多
- 拖拽次数变多
- 右侧栏变得冗长
- 容易误操作

那就不应该继续往下拆。

默认应优先选择：

**对用户来说最自然的最小编辑单位**

而不是对开发者最省事的最小实现单位。

### 2.3 导入适配优先

后续大量外来组件不会以“完美符合本系统结构”的方式出现。

因此我们需要一种能稳定落库的目标层级：

- 原语归原语
- 控件归控件
- 壳归壳
- 成品卡先拆，再进入资产系统

这样以后导入来源变化时，底层方法不变。

## 3. 分层总览

`Kit Studio` 的组件体系统一分成五层：

### P0 内置原语

系统内部复用的最小视觉/交互单元。

### P1 原子控件

用户可直接拖入画布的最小业务控件。

### P2 复合控件

对用户仍是一个控件，但内部由多个原语或子结构组成。

### P3 Card Shell

真正的容器层，负责边界、slot、内部控件承接，不负责具体业务语义。

### P4 成品卡 / 导入卡

从外部来源或内部模板沉淀出的完整卡片资产，进入系统前必须先拆层，不直接作为黑盒永久保留。

---

## 4. 各层定义

## 4.1 P0 内置原语

### 定义

只服务视觉和局部交互，不直接表达业务语义的内部构件。

### 特征

- 不出现在左侧资产栏
- 不支持单独拖入画布
- 不单独选中
- 不单独拥有右侧属性面板
- 只能依附于控件、卡壳、编辑器面板存在

### 典型示例

- `close icon`
- `copy icon`
- `drag handle`
- `resize handle`
- `caret / chevron`
- `badge dot`
- `checkbox indicator`
- `radio indicator`
- `input prefix icon`
- `collapse trigger`

### 设计规则

P0 原语可以在代码里被高度复用，但不能直接暴露成业务控件。

否则会导致：

- 左栏资产污染
- 用户误以为它能独立表达业务
- 导入组件时拆得过碎
- 右侧栏需要处理无意义配置

---

## 4.2 P1 原子控件

### 定义

用户能直接拖入卡片或画布，且本身就有明确业务语义的最小控件。

### 特征

- 出现在左侧 `Controls`
- 可拖拽、选中、删除、复制
- 右侧栏有独立配置
- 不承接自由子控件
- 不负责内部 nested canvas

### 典型示例

- `Heading`
- `Text`
- `Button`
- `Text Input`
- `Number Input`
- `Textarea`
- `Select`
- `Checkbox`
- `Radio`
- `Divider`

### 判断标准

满足以下条件时，定义为 P1：

1. 单独放入卡片中，用户可以理解它的业务意义
2. 右侧栏字段清晰、有限
3. 用户通常不会要求再把它内部拆成多个可拖单位

### 说明

`Checkbox` 的最小编辑单位应是“一个完整复选项”，而不是方框本身。

也就是说：

- 小方框 + 对勾图形 = P0 原语
- 完整单项勾选控件 = P1 原子控件

---

## 4.3 P2 复合控件

### 定义

内部结构不止一个原子单元，但对用户来说仍然应该作为一个整体来使用和配置。

### 特征

- 对外仍是一个控件
- 一个选中框
- 一个拖拽体
- 一个右侧栏
- 内部不开放自由拖拽组装
- 可以有内部数据结构或 slot，但不变成自由卡壳

### 典型示例

轻型复合控件：

- `Checkbox Group`
- `Radio Group`
- `Search Input`
- `Info Row`
- `User Identity Row`
- `Tag List`

重型复合控件：

- `Table`
- `Chart`
- `Calendar`
- `Stat Block`
- `Kanban Summary`
- `Rich Metric Card`

### 为什么要单列 P2

因为很多组件既不是简单原子控件，也不该默认升级为卡壳。

例如：

- 图表内部有坐标轴、图例、tooltip、series
- 表格内部有表头、行、列、分页、排序

这些结构确实复杂，但用户大多数时候不是要把它们拆开拖来拖去，而是要：

- 设数据源
- 设字段映射
- 设样式
- 设交互

因此它们应作为 **大型单体控件** 存在，也就是 P2 的重型子类。

### P2 的默认策略

除非明确需要自由布局，否则不要升级为卡壳。

优先形态是：

**一个控件 + 一套结构化右侧栏 + 必要的内部 slot**

---

## 4.4 P3 Card Shell

### 定义

只承担结构边界、slot、容器、内部控件承接职责的卡壳。

### 特征

- 能接收其他控件
- 具备 nested canvas
- 有 header / footer / padding / gap / scrollbar 等壳层属性
- 不直接表达具体业务内容

### 典型示例

- `Blank Card Shell`
- `Panel`
- `Canvas Slot`
- 后续各种业务卡的结构壳

### 当前仓库中的映射

目前代码中的 `panel`，产品语义上就是 `Card Shell`。

### 规则

只有当内部需要：

- 接收多个控件
- 子控件独立选中
- 子控件独立拖拽
- 子控件自由排序
- 子控件单独绑定/动作

时，才应该使用 P3。

不要因为一个组件看起来“视觉上像一张卡”，就直接定义成 Card Shell。

---

## 4.5 P4 成品卡 / 导入卡

### 定义

来源于外部库或内部模板库的完整业务卡片资产。

### 特征

- 外观完整
- 语义完整
- 可能包含壳、内容、按钮、标题、数据区
- 进入系统前必须先做结构拆解

### 典型示例

- `shadcn login card`
- `author card`
- `API info card`
- `stats summary card`
- `feature hero card`

### 规则

P4 不是最终运行时底层类型，而是导入和资产沉淀阶段的来源形态。

它进入平台后，必须拆成：

1. `Card Shell`
2. 内部 `P1 / P2 控件`
3. `P0 原语`
4. 数据结构
5. 可编辑 tokens
6. Inspector schema

也就是：

**P4 是进口形态，不是最终底层语义。**

---

## 5. 分类判断流程

后续开发和导入适配时，按以下顺序判断：

### 第一步：它能否独立表达业务语义

如果不能，直接归为 `P0 原语`。

例如：

- X 关闭
- 拖动柄
- 下拉箭头
- 勾选图标

### 第二步：它是否应该被用户直接拖入画布

如果应该，且本身语义单一清晰，则归为 `P1 原子控件`。

### 第三步：它是否内部复杂，但对用户仍应整体使用

如果是，则归为 `P2 复合控件`。

### 第四步：它内部是否需要承接其他独立控件

如果是，则归为 `P3 Card Shell`。

### 第五步：它是否是外来成品卡

如果是，则先标记为 `P4`，进入拆解流程，不直接按黑盒投放。

## 6. 从开发搭建角度的最优分类

为了降低后续开发成本，底层实现需要优先满足以下目标：

### 6.1 原语复用

所有：

- 图标按钮
- 勾选标记
- 收起箭头
- 拖动柄
- 复制按钮

都应抽为统一原语，而不是每个控件内部各写一份。

### 6.2 原子控件保持窄而稳

不要把 `Text`、`Heading` 做成万能容器。

例如不应把：

- 图标行
- 状态块
- 接口信息块
- 带按钮的提示条

都强行塞进 `Text`。

否则：

- 右侧栏会失控
- 语义会混乱
- 导入适配时无法稳定落层

### 6.3 复合控件承担“一个完整功能块”

这类控件最适合吸收社区成熟部件。

例如：

- `search box with icon`
- `checkbox group`
- `metric tile`
- `table`
- `chart`

内部可用多个原语和布局结构实现，但对系统外层保持单体。

### 6.4 卡壳只管结构

`Card Shell` 不负责塞业务语义，不负责变成复杂超级控件。

它只负责：

- 容器边界
- header / footer
- padding / gap
- scroll policy
- nested canvas

---

## 7. 从用户效率角度的最优分类

为了让用户在组装时效率最高，需要优先避免两种错误：

### 错误一：拆得太细

表现：

- 用户要选中很多次
- 一个业务块要拖很多小零件
- 右侧栏到处重复
- 容易误删、误拖、误选

### 错误二：做得太黑盒

表现：

- 看起来像卡，但不能调关键内容
- 只是静态模板，不像真正资产
- 无法映射数据、动作、主题

### 用户侧最佳平衡点

#### 应该直接暴露给用户的

- 标题
- 段落
- 按钮
- 输入框
- 单个复选项
- 分割线
- 卡壳
- 表格
- 图表

#### 不应该直接暴露给用户的

- 图标关闭按钮
- 复制图标
- 折叠箭头
- 输入框前缀图标
- Checkbox 小方框本体

#### 应该作为整体暴露，而不是继续拆开的

- 复选组
- 搜索输入框
- 接口信息块
- 用户信息条
- 图表
- 表格

---

## 8. 对右侧栏分组的直接影响

组件分层会直接决定 inspector 的大组结构。

## 8.1 P0 原语

不拥有独立右侧栏。

其配置应挂在宿主控件或宿主卡壳里。

例如：

- 关闭图标样式属于按钮或卡壳配置
- checkbox indicator 样式属于 checkbox 控件配置

## 8.2 P1 / P2 控件

统一使用以下大组：

1. `Content`
2. `Layout`
3. `Appearance`
4. `Data`
5. `Logic`

说明：

- `P1` 的 `Content` 往往较窄
- `P2` 的 `Data` 往往更重

例如 `Chart`：

- `Content`：标题、副标题、空态文案
- `Layout`：尺寸、占位、高度策略
- `Appearance`：主题、边框、圆角、颜色、字体
- `Data`：数据源、字段映射、series 配置、聚合
- `Logic`：点击、联动、刷新、跳转

## 8.3 P3 Card Shell

统一使用以下大组：

1. `Shell`
2. `Layout`
3. `Appearance`
4. `Data`
5. `Logic`

其中：

- `Shell`：Header / Footer / Overflow / Slot Policy
- `Layout`：Size / Padding / Gap / Auto Height
- `Appearance`：Typography / Border / Corner / Children follow
- `Data`：默认数据上下文、子控件继承策略、后续 slot contract
- `Logic`：Bindings / Actions / Handoff / Constraints

## 8.4 P4 成品卡

不建议长期保留“大一统右侧栏”。

正确做法是：

- 选中卡壳时，显示卡壳面板
- 选中内部控件时，显示控件面板

也就是：

**卡壳管卡壳，控件管控件，不把所有内容捏在一个面板里。**

## 9. 外来成品卡的标准拆解法

外来卡片进入系统时，统一按以下顺序拆解：

### 9.1 先识别壳边界

确认最外层是否存在：

- 明确边框
- 背景块
- padding 边界
- header / footer / body

若存在，先拆出 `P3 Card Shell`。

### 9.2 再识别内部业务单元

识别哪些部分对用户来说应该是独立可编辑内容：

- 标题
- 文本
- 操作按钮
- 输入框
- 标签组
- 图表
- 表格

这些落为 `P1` 或 `P2`。

### 9.3 再识别附属原语

以下内容通常不应落成控件：

- 图标
- 收起箭头
- 复制按钮
- 小红点
- 尾部状态点
- 勾选图形

这些归为 `P0`。

### 9.4 重复项优先做数据结构，不优先做独立控件

例如：

- table rows
- select options
- checkbox options
- tab items
- menu items

默认应视为数据项，而不是一个个可拖拽控件。

只有在明确要求：

- 每项独立拖拽
- 每项独立动作
- 每项独立布局
- 每项独立主题

时，才升级为控件。

### 9.5 大型复杂模块默认收口为单体控件

以下默认优先归为 `P2`：

- `Table`
- `Chart`
- `Calendar`
- `Kanban`
- `Rich Text`

不要一导入就拆成自由卡壳。

### 9.6 只在必要处开放 slot

当用户确实需要局部自由编排时，再在大型控件上开放 slot。

例如：

- `Table.toolbar`
- `Table.emptyState`
- `Chart.header`
- `Chart.footer`

而不是把整张表和整张图直接拆散。

## 10. 典型案例

## 10.1 单个复选项

正确拆法：

- 小方框 = `P0`
- 对勾图形 = `P0`
- 完整复选项 = `P1 Checkbox`

## 10.2 复选组选项列表

正确拆法：

- 每个 option 默认是内部数据项
- 整组 = `P2 Checkbox Group`

只有在每项需要独立布局时，才继续升级。

## 10.3 搜索框带图标

正确拆法：

- 搜索图标 = `P0`
- 输入体 = 内部结构
- 整体 = `P2 Search Input`

不要把它拆成一个图标控件加一个文本输入控件让用户手动拼。

## 10.4 图表

正确拆法：

- chart body = `P2`
- legend / tooltip / axis 默认都是内部结构
- 有明确配置项，但不是独立画布控件

## 10.5 表格

正确拆法：

- table = `P2`
- row / cell / header 默认不是独立控件
- toolbar / empty state 可按需要扩展 slot

## 10.6 外部导入的 API 信息卡

类似：

- 标题
- 域名徽标
- 地址文本
- 复制按钮

建议拆法：

1. 最外层边界 = `P3 Card Shell`
2. 标题区信息 = `P2 Info Header`
3. 地址展示 = `P1 Text` 或 `P2 Endpoint Display`
4. 复制按钮 = 内部原语或附属动作按钮

是否把复制按钮升为独立控件，要看用户是否真的需要在组装时频繁替换和移动它。

默认不升。

## 12. 禁止事项

后续开发中，尽量避免以下做法：

### 12.1 把文本控件做成万能容器

这会让：

- 语义失真
- Inspector 失控
- 外来组件无法稳定映射

### 12.2 把所有复杂块都做成 Card Shell

这会导致：

- 用户组装成本变高
- 内部拖拽复杂度上升
- 导入适配难度增加

### 12.3 把所有小元素都暴露成控件

这会导致左栏和右侧栏碎片化。

### 12.4 让导入卡保持黑盒整体

这会破坏：

- 属性面板一致性
- 主题 token 统一
- AI handoff 清晰度
- 导出代码可维护性

---

## 13. 验收清单

一个新组件或导入组件进入系统前，至少要完成以下判断：

1. 它属于 `P0 / P1 / P2 / P3 / P4` 的哪一层
2. 它是否应该出现在左侧资产栏
3. 它是否应独立拥有右侧栏
4. 它是否需要 nested canvas
5. 它的最小业务语义单位是什么
6. 它内部哪些是数据项，哪些是控件
7. 它内部哪些是原语
8. 它的右侧栏应落到哪个大组
9. 它导入后是否需要拆壳
10. 它导出后是否仍然保持语义清晰

只有这 10 项都能说清，才允许进入正式资产库。

---

## 14. 最终结论

`Kit Studio` 后续组件体系的稳定方法，不是把所有东西都拆成最小代码单元，而是把它们拆成：

- 对系统最稳定
- 对用户最高效
- 对导入最可复用

的最小业务单元。

因此后续统一采用：

- `P0 原语`
- `P1 原子控件`
- `P2 复合控件`
- `P3 Card Shell`
- `P4 成品卡 / 导入卡`

这五层作为平台底层分类基线。

任何新增控件、卡壳、外来组件适配、右侧栏重构，都必须先回到这五层判断。
